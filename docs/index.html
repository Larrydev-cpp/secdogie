$ drive any screen

secdogie points a vision LLM at a screenshot, gets back one action, executes it, and repeats — on your desktop, an Android phone, or an iPhone. Plus an encrypted tunnel to reach those machines and a multi-model reader for 3D scenes.

Get started View source → 
$ git clone https://github.com/larrydev-cpp/secdogie
$ cd secdogie/agent && pip install -e .
$ secdogie-agent "open the notes app and write my standup"

The pieces

One agent loop, many things to point it at

Every tool here shares the same screenshot → model → action loop. Only the backend — how a frame is captured and an action carried out — changes. Each ships as its own CLI and its own single-file build.

secdogie-agent

desktop

The core: screenshots your machine, asks a vision model for the next click/type/scroll, and does it — with a confirmation on every step until you trust it.

$ secdogie-agent "task" --dry-run

secdogie-android

adb

Same loop over the Android Debug Bridge — screencap in, input out. Nothing installed on the phone. Re-targets taps by UI element, not by pixel.

$ secdogie-android "task" --device SERIAL

secdogie-ios

webdriveragent

Drives an iPhone or iPad through a running WebDriverAgent over plain HTTP — the same action schema, translated to WDA gestures.

$ secdogie-ios "task" --wda-url ...

secdogie-open

local web ui

A local web page (this very design system) that splits your screen into regions and runs the agent across several windows at once — one board, many jobs.

$ secdogie-open # opens in your browser

secdogie-scene3d

multi-model

N workers each analyze one camera view of a 3D scene concurrently, spread over a pool of API keys; an aggregator fuses their observations into one layout.

$ secdogie-scene3d front=a.png top=b.png ...

secdogie-tunnel

C · libsodium

An encrypted UDP tunnel (XChaCha20-Poly1305, Noise-style handshake, replay window) with a hub mode that terminates many client tunnels and routes by inner IP.

$ secdogie-tunnel hub hub.conf
RPA macros are built in. Add --macro run.json to the agent or the Android tool: the first successful run is recorded and replayed later with zero model calls, re-finding each target by element identity and falling back to the live model the moment the UI changes.

How it fits

The same four moves, every platform

  • 01
    A Backend abstraction does the platform work.Desktop (mss + pyautogui), adb, or WebDriverAgent all implement setup / capture / execute. The loop never knows which one it's driving.
  • 02
    A VisionProvider does the thinking.Anthropic and OpenAI sit behind one interface returning a single JSON action. The model id picks the provider; adding another is one subclass.
  • 03
    Coordinates are always mapped back.Screenshots are downscaled for the model, then its coordinates are scaled to real pixels — so clicks land where the model intends.
  • 04
    You stay the backstop.Every non-benign action asks for a y/N first unless you pass --auto; --dry-run reasons and logs without ever touching the machine.

Start

Running in three commands

You need Python 3 and an API key for one provider (Anthropic for claude-*, OpenAI for gpt-*). No account, no cloud, nothing to sign up for beyond the model key.

1 · install

Clone & install

$ git clone …/secdogie
$ cd secdogie/agent
$ pip install -e .

2 · key

Point it at a key

$ secdogie-agent --init-config
# edit the file, paste your key
# or: export ANTHROPIC_API_KEY=…

3 · run

Give it a task

$ secdogie-agent \
    "summarize the open PDF" \
    --dry-run
No Python on the target? Every component builds to a single self-contained executable with PyInstaller (packaging/build.sh or build.ps1), and the full walkthrough lives in TUTORIAL.md.
